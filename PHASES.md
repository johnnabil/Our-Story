# Build Phases — Our Story

This document breaks the project into ordered build phases. Each phase is self-contained — complete it fully before moving on. Every phase lists the exact files to create, acceptance criteria to verify, and dependencies on previous phases.

---

## Phase 0 — Project Scaffold
**Goal**: Working Next.js 15 app with Tailwind v4 that renders a blank page.

### Files
| File | Action |
|------|--------|
| `package.json` | Create — all dependencies listed |
| `tsconfig.json` | Generated by Next.js, verify `strict: true` |
| `next.config.ts` | Create — Cloudinary remote image patterns |
| `postcss.config.mjs` | Create — `@tailwindcss/postcss` plugin |
| `app/globals.css` | Create — `@import "tailwindcss"` + `@theme` with all color/font tokens |
| `app/layout.tsx` | Create — load fonts, apply CSS variables to `<body>`, minimal HTML shell |
| `app/page.tsx` | Create — return a `<main>` with "Hello" heading |
| `.env.local.example` | Create — all env var placeholders |
| `vercel.json` | Create |

### Acceptance Criteria
- [ ] `bun install` completes without errors
- [ ] `bun run dev` starts on localhost:3000
- [ ] Page renders "Hello" with Cormorant Garamond heading and DM Sans body text
- [ ] Tailwind utility classes work (e.g., `bg-cream` applies the correct color)

### Dependencies
None — this is the starting phase.

## Phase 1 — Types & Data Layer
**Goal**: Type system + Redis helpers + default content. No UI changes yet.

### Files
| File | Action |
|------|--------|
| `lib/types.ts` | Create — all interfaces, `ContentKey`, `CONTENT_KEYS` |
| `lib/kv.ts` | Create — `getContent`, `setContent`, `getAllContent` using `@upstash/redis` |
| `data/defaults/<key>.json` | Create — realistic placeholder content split across all 8 keys |

### Acceptance Criteria
- [ ] `bun run build` passes with no TypeScript errors
- [ ] Split defaults conform to each `SiteContent` key shape
- [ ] If `UPSTASH_REDIS_REST_URL` is not set, `getAllContent` gracefully returns defaults

### Dependencies
Phase 0 (TypeScript config, dependencies installed).

---

## Phase 2 — Auth
**Goal**: NextAuth v5 configured, login/logout working.

### Files
| File | Action |
|------|--------|
| `lib/auth.ts` | Create — NextAuth config, credentials provider, JWT 7-day session |
| `app/api/auth/[...nextauth]/route.ts` | Create — export handlers |

### Acceptance Criteria
- [ ] `bun run build` passes
- [ ] `POST /api/auth/callback/credentials` with correct password returns a session cookie
- [ ] `POST /api/auth/callback/credentials` with wrong password returns error
- [ ] `GET /api/auth/session` returns the session when authenticated

### Dependencies
Phase 1 (types for session user shape).

---

## Phase 3 — API Routes
**Goal**: All content CRUD + Cloudinary upload/destroy endpoints working.

### Files
| File | Action |
|------|--------|
| `lib/cloudinary.ts` | Create — `generateUploadSignature`, `destroyImage` |
| `app/api/content/route.ts` | Create — `GET` all content |
| `app/api/content/[key]/route.ts` | Create — `PATCH` single key (auth-gated) |
| `app/api/upload/route.ts` | Create — `POST` Cloudinary signature (auth-gated) |
| `app/api/image/[publicId]/route.ts` | Create — `DELETE` Cloudinary image (auth-gated) |

### Acceptance Criteria
- [ ] `GET /api/content` returns full `SiteContent` JSON (from split defaults if Redis empty)
- [ ] `PATCH /api/content/hero` without auth → 401
- [ ] `PATCH /api/content/hero` with auth + valid body → 200, data persists in Redis
- [ ] `PATCH /api/content/invalid` → 400
- [ ] `POST /api/upload` returns valid Cloudinary signature params
- [ ] `DELETE /api/image/test-id` calls Cloudinary destroy (verify no 500)

### Dependencies
Phase 2 (auth for session validation).

---

## Phase 4 — Providers
**Goal**: Client-side context for content state and edit mode.

### Files
| File | Action |
|------|--------|
| `components/providers/ContentProvider.tsx` | Create — fetch, state, optimistic update, debounce, flushAll |
| `components/providers/EditProvider.tsx` | Create — isEditing toggle, canEdit from session |
| `app/layout.tsx` | Modify — wrap children with `SessionProvider > ContentProvider > EditProvider` |

### Acceptance Criteria
- [ ] Page loads → ContentProvider fetches `/api/content` → content is available via `useContent()` hook
- [ ] `updateContent("hero", newHero)` updates state immediately and PATCHes after 300ms
- [ ] `flushAll()` fires all pending patches immediately
- [ ] `useEdit()` returns `{ isEditing, toggleEdit, canEdit }`
- [ ] Toggling edit when not authenticated triggers sign-in

### Dependencies
Phase 3 (API routes must exist for ContentProvider to call).

---

## Phase 5 — Editable Components
**Goal**: All 5 inline-editing primitives working in isolation.

### Files
| File | Action |
|------|--------|
| `components/edit/EditableText.tsx` | Create |
| `components/edit/EditableImage.tsx` | Create |
| `components/edit/EditableDate.tsx` | Create |
| `components/edit/EditableTags.tsx` | Create |
| `components/edit/EditableRichText.tsx` | Create |

### Acceptance Criteria
- [ ] In edit mode, each component renders its interactive variant
- [ ] In view mode, each component renders read-only markup
- [ ] `EditableText`: strips HTML on paste, fires onChange on blur with plain text
- [ ] `EditableImage`: opens file picker, uploads to Cloudinary, deletes old image, fires onChange
- [ ] `EditableDate`: date input in edit mode, formatted string in view mode
- [ ] `EditableTags`: add via +, remove via ×, fires onChange with updated array
- [ ] `EditableRichText`: textarea in edit mode, `<p>` with pre-wrap in view mode

### Dependencies
Phase 4 (needs `useEdit()` context to know if editing).

## Phase 6 — UI Components
**Goal**: EditBar, Modal, and Nav fully functional.

### Files
| File | Action |
|------|--------|

| `components/ui/Modal.tsx` | Create — backdrop, focus trap, Escape close |
| `components/ui/Nav.tsx` | Create — sticky top nav, smooth scroll links, backdrop blur |

### Acceptance Criteria
- [ ] EditBar visible bottom-right on all viewport sizes
- [ ] EditBar shows "Edit" when not authed, "Edit Site" when authed, editing controls when editing
- [ ] "Save All" calls `flushAll()` from ContentProvider
- [ ] Modal opens/closes, traps focus, closes on Escape and backdrop click
- [ ] Nav links scroll smoothly to section anchors
- [ ] Nav has blur backdrop on scroll

### Dependencies
Phase 4 + 5 (EditBar uses providers, Modal may be used by editable components).

## Phase Summary

```
Phase 0  Scaffold          → blank page runs
Phase 1  Types + Data      → data model defined, Redis helpers ready
Phase 2  Auth              → login/logout works
Phase 3  API Routes        → all CRUD endpoints tested
Phase 4  Providers         → content state + edit mode available to components
Phase 5  Edit Components   → 5 inline-edit primitives working
Phase 6  UI Components     → EditBar, Modal, Nav functional
Phase 7  Sections          → all 7 sections rendered + editable
Phase 8  Polish + Deploy   → production-ready, deployed
```

Each phase should take 15–45 minutes for an AI agent, 30–90 minutes for a human developer.
